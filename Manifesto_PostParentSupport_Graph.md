# Manifesto: Post-Parent Support & Advocacy Graph
# 親亡き後・親ある内の権利擁護支援マニフェスト

**Version:** 3.0 (Complete Edition)
**Last Updated:** 2025-12-08

---

## 0. 理念 (Philosophy)

我々が構築するのは、単なる「障害者情報データベース」ではない。
親たちが長い時間をかけて蓄積してきた**「我が子を守るための暗黙知（Wisdom）」**と**「愛（Care）」**を、親亡き後も機能する**「社会的なシステム」**へと継承するためのデジタル・アーカイブである。

このナレッジグラフは、以下の5つの価値を最優先する。

1.  **Dignity (尊厳):** 管理対象としてではなく、歴史と意思を持つ一人の人間として記録する。
2.  **Safety (安全):** 緊急時に「誰が」「何を」すべきか、迷わせない構造を作る。
3.  **Continuity (継続性):** 支援者が入れ替わっても、ケアの質と文脈を断絶させない。
4.  **Resilience (強靭性):** 親が倒れた際、その機能を即座に代替できるバックアップ体制を可視化する。
5.  **Advocacy (権利擁護):** 本人の声なき声を拾い上げ、法的な後ろ盾と紐づける。

## 1. データモデルの5本柱 (The 5 Pillars)

システムは以下の5つの領域（ドメイン）によって構成される。

### 第1の柱：本人性 (Identity & Narrative)
* **役割:** 「その人は誰か」を定義する。属性だけでなく、人生の物語を含む。
* **主要ノード:** `:Client` (本人), `:LifeHistory` (生育歴), `:Wish` (本人・親の願い)

### 第2の柱：ケアの暗黙知 (Care Instructions)
* **役割:** 「どう接すべきか」を定義する。親の頭の中にあったマニュアルを形式知化する。
* **主要ノード:** `:CarePreference` (具体的配慮), `:NgAction` (禁忌事項), `:Condition` (特性)

### 第3の柱：危機管理ネットワーク (Safety Net)
* **役割:** 「誰が守るか」を定義する。緊急時の指揮命令系統と法的権限。
* **主要ノード:** `:KeyPerson` (キーパーソン), `:Guardian` (成年後見人), `:Hospital` (医療機関)

### 第4の柱：法的基盤 (Legal Basis)
* **役割:** 「何の権利があるか」を定義する。支援を受けるための資格と行政の決定。
* **主要ノード:** `:Certificate` (手帳), `:PublicAssistance` (保護), `:Organization` (行政機関)

### 第5の柱：親の機能と移行 (Parental Transition)
* **役割:** 「親が担っている機能は何か」を定義し、その移転先を管理する。
* **主要ノード:** `:Relative` (親族), `:CareRole` (親のタスク), `:Service` (代替サービス)

## 2. スキーマ定義 (Schema Definition)

AIおよび開発者は、以下のスキーマ定義を厳守すること。

### 2.1 ケアと配慮 (Care & Instructions)
現場の支援者が最も必要とする「取り扱い説明書」。

| ノードラベル | プロパティ例 | 説明 |
| :--- | :--- | :--- |
| **:CarePreference** | `category` (食事/睡眠/パニック), `instruction`, `priority` (High/Medium) | 具体的な介助手順。「肉は一口大に切る」「雷が鳴ったら音楽を流す」等。 |
| **:NgAction** | `action`, `reason`, `riskLevel` (LifeThreatening/Panic) | **絶対にしてはいけないこと。** (例: 背後から触れる、特定の薬剤投与)。 |
| **:Condition** | `name`, `diagnosisDate` | 医学的診断名（自閉スペクトラム症、てんかん等）。 |

* **リレーション:**
    * `(:Client)-[:REQUIRES]->(:CarePreference)`
    * `(:Client)-[:MUST_AVOID]->(:NgAction)`

### 2.2 親の機能不全と代替 (Parental Risks & Backup)
親の健康リスクと、親が担う役割のバックアップ。

| ノードラベル | プロパティ例 | 説明 |
| :--- | :--- | :--- |
| **:Relative** | `healthStatus` (Healthy/Declining/Hospitalized), `age` | 親の基本情報と現在の健康状態。 |
| **:CareRole** | `name` (金銭管理, 送迎, 料理) | 親が現在行っているタスク。 |

* **リレーション:**
    * `(:Relative)-[:PERFORMS]->(:CareRole)` (親がその役割を担っている)
    * `(:CareRole)-[:CAN_BE_PERFORMED_BY]->(:Service)` (代替サービスがある)
    * `(:CareRole)-[:CAN_BE_PERFORMED_BY]->(:Supporter)` (代替人物がいる)

### 2.3 危機管理と権限 (Emergency & Authority)
親亡き後、あるいは親の緊急搬送時の意思決定プロセス。

| ノードラベル | プロパティ例 | 説明 |
| :--- | :--- | :--- |
| **:KeyPerson** | `rank` (1, 2...), `role` (緊急連絡先, 医療同意) | 緊急時に連絡する優先順位付きの人物。 |
| **:Guardian** | `type` (成年後見/保佐/補助/任意), `name` | 法的代理権を持つ人物・法人。 |

* **リレーション:**
    * `(:Client)-[:HAS_KEY_PERSON {rank: 1}]->(:KeyPerson)`
    * `(:Client)-[:HAS_LEGAL_REP]->(:Guardian)`

### 2.4 ナラティブとパスポート (Narrative & Passport)
従来の基本情報。

| ノードラベル | プロパティ例 | 説明 |
| :--- | :--- | :--- |
| **:Client** | `name`, `dob`, `bloodType` | 本人情報。 |
| **:LifeHistory** | `era`, `episode` | 生育歴。「何が好きで何が嫌いだったか」の背景。 |
| **:Wish** | `content`, `status` (Active/Archived) | 「地域で暮らしたい」等の希望。 |
| **:Certificate** | `type`, `grade`, `nextRenewalDate` | 手帳・受給者証。更新期限の管理に使用。 |

## 3. AI運用プロトコル (AI Operational Protocol)

AIアシスタント（Claude等）は、以下の優先順位に従って回答を生成すること。

### ルール1：安全第一 (Safety First)
ユーザーから「パニック」「事故」「急病」を示唆する入力があった場合、AIは直ちに以下の情報を最優先で検索・提示しなければならない。
1.  **禁忌事項 (:NgAction)** - 二次被害を防ぐため。
2.  **具体的対処 (:CarePreference)** - その場を落ち着かせるため。
3.  **緊急連絡先 (:KeyPerson)** - ランク1位の人物。

### ルール2：親の機能不全トリガー (Parent Down Trigger)
「母が入院した」「父が認知症になった」という入力があった場合、AIは直ちに以下を実行すること。
1.  その親(`:Relative`)が担っていたタスク(`:CareRole`)を特定する。
2.  そのタスクに紐づく代替手段(`:Service`, `:Supporter`)を提示する。
3.  代替手段が未登録の場合は、「緊急の支援調整が必要です」と警告する。

### ルール3：文脈の尊重 (Context Awareness)
新しい支援先を探す際、単に「空きがある施設」を探すのではなく、本人の **:LifeHistory（過去の成功体験・失敗体験）** と **:Condition** に適合するかを照合（マッチング）すること。

## 4. AIプロンプトテンプレート (System Prompt for Data Entry)

データを登録する際は、以下のテンプレートを使用し、日常会話から構造化データへ変換する。

```text
あなたは「親亡き後支援データベース」のデータ構築エンジニアです。
提供されたテキスト（親からの聞き取りや相談記録）から、以下のルールでCypherクエリを生成してください。

1. **暗黙知の抽出:** 「～すると落ち着く」「～は嫌がる」という記述を見つけたら、必ず :CarePreference または :NgAction ノードとして切り出すこと。
2. **親の機能分解:** 「毎朝、母が薬を管理している」という記述があれば、母(:Relative)が「服薬管理」という(:CareRole)を担っている構造を作成すること。
3. **緊急性の定義:** 連絡先人物が含まれる場合、その人が「緊急連絡先」なのか「法的代理人」なのかを区別すること。
4. **日付の記録:** すべての日付プロパティには `date()` を使用すること。

【入力テキスト例】
「息子は大きな音が苦手で、パニックになると自分の頭を叩きます。その時は静かな部屋で背中をさすると落ち着きます。
現在は私が全ての食事を作っていますが、私は来月から入院手術の予定があります。もしもの時は、叔父の田中さん（090-xxxx）に連絡してください。」
