# 安全性コンプライアンスチェック（Phase 3 サブステップ）

> `lib/ai_extractor.py::SAFETY_CHECK_PROMPT` の skill 版移植。
> 新規支援記録やケア行動が、既存の禁忌事項（NgAction）に抵触していないかを判定する。

---

## 役割

あなたは「親亡き後支援データベース」の安全管理責任者です。
入力された「ナラティブ（支援記録中の行動）」と、Neo4j に登録されている当該クライアントの「禁忌事項（NgAction）」を照合し、**安全性違反（コンプライアンス違反）**がないかを判定してください。

---

## 判定プロセス

### Step 1: 既存 NgAction の取得

Phase 3a で取得した `[{action, riskLevel, reason}, ...]` のリストを入力として受け取る。

### Step 2: 行動の照合

ナラティブ中に現れる「行動」「介入」「対応」と、既存 NgAction の `action` を**意味レベル**で照合する。

**判定基準**:

| ケース | 判定 |
|---|---|
| 禁忌行動と完全一致 | `is_violation: true`, `risk_level: 既存のriskLevelをそのまま` |
| 禁忌行動と類似（同一カテゴリ） | `is_violation: true`, `risk_level: 1段階下げて表示` |
| 禁忌に関連するが実施者が防止策を取っている | `is_violation: false`, `warning: 注意喚起のみ` |
| 無関係 | `is_violation: false` |

**類似判定の例**:
- 禁忌「ピーナッツアレルギー」 vs ナラティブ「ナッツ類を提供」→ **違反**
- 禁忌「大音量パニック」 vs ナラティブ「ライブ会場に連れて行った」→ **違反**
- 禁忌「拘束禁止」 vs ナラティブ「暴れたので手を握り続けた」→ **要確認**（表現次第）

### Step 3: 出力

以下の JSON 形式で出力:

```json
{
  "is_violation": true,
  "violations": [
    {
      "ng_action": "突然の大きな音",
      "matched_narrative": "工事音でパニック",
      "severity": "High",
      "recommendation": "該当の状況が既に発生しています。このログ登録は事実記録として可ですが、今後は事前に工事予定の確認を CarePreference に追加することを推奨します。"
    }
  ],
  "warning_message": "深刻な違反の疑いがあります。禁忌事項「突然の大きな音」に該当する状況が記録されています。",
  "risk_level": "High"
}
```

**違反なしの場合**:

```json
{
  "is_violation": false,
  "violations": [],
  "warning_message": null,
  "risk_level": "None"
}
```

---

## 重要な区別

### 事後記録 vs 事前承認

このチェックは主に**登録内容が矛盾していないか**の検証であり、事後記録の場合は「違反が発生した事実」を記録することは適切です。ただし以下は**必ずユーザーに警告**:

1. **禁忌行動を介入手段として記録しようとしている場合**（例: 禁忌「拘束」を support action として登録）
2. **LifeThreatening レベルの禁忌に抵触**した場合（必ず KeyPerson / Guardian への共有を促す）
3. **複数の禁忌に同時抵触**している場合

### 新規 NgAction の追加提案

ナラティブから新たな禁忌候補が見つかった場合（例: 今回初めて「大きな音でパニック」が観察された）、既存 NgAction と重複していなければ、抽出フェーズで新規 NgAction ノードとして生成する。本チェックでは**重複登録を防ぐ**役割も担う。

---

## 出力先

判定結果は Phase 3c のプレビューテキスト内に以下のように埋め込む:

```
▼ ⚠️ 安全性チェック結果
  既存禁忌事項: 3件
  抵触判定: 1件の違反候補
  ─ 禁忌「突然の大きな音」に抵触（リスク: High）
     該当箇所: "工事音でパニック"
     推奨対応: 事前の工事予定確認を CarePreference に追加
```

ユーザーが登録を継続する場合、AuditLog の `details` フィールドに違反判定の結果を必ず含めること。
