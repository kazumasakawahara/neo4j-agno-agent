# 現場・管理連携型 知的障害者支援システム 実装計画

> **注意:** 本計画は開発ロードマップであり、実際のコード記述は行いません。実装を進める際は、各タスクの指示に従ってください。

**目標:** グループホーム・B型作業所の現場での「ハイブリッド感情ログ」と「AI相談」を、管理者の「リスク予測」に直結させるシステムの構築。

**アーキテクチャ:** 
- 既存のStreamlit UIを拡張し、モバイル向け「現場モード」とPC向け「管理モード」を実装。
- Neo4jの `:SupportLog` ノードを拡張して感情・タグ・文脈を保存。
- バックエンドで移動平均を用いた統計的なリスク予測ロジックを稼働させる。

**技術スタック:** Python, Streamlit, Neo4j, Gemini Embedding 2, Claude Desktop Skills

---

## 1. 開発フェーズの概要

本プロジェクトは以下の4つのフェーズで構成されます。

### フェーズ1：データ基盤の拡張
- `:SupportLog` ノードへの新プロパティ（emotion, triggerTag, context等）の追加。
- 既存データの移行（バックフィル）計画。

### フェーズ2：現場スタッフ用UI（モバイル最適化）
- 3ステップのハイブリッド入力フローの実装。
- AI相談ボタンとチャットインターフェースの統合。
- 禁忌事項（NgAction）の常時表示エリアの作成。

### フェーズ3：管理者用ダッシュボード（分析・通知）
- リアルタイム・インサイト・フィードの実装。
- リスク予測エンジンの構築（頻度変化、複合リスク、スタッフ負担の3軸）。
- アラート通知UIの実装。

### フェーズ4：統合テストと現場検証
- ダミーデータを用いた予測ロジックの精度検証。
- 現場スタッフ・管理者による操作性テスト。

## 2. 詳細タスク一覧

### タスク1: Neo4j スキーマの更新
**目的:** 新しいログ形式を保存できるようにする。
- `:SupportLog` に `emotion` (Joy/Anger/Sorrow), `triggerTag` (Meal/Work/etc), `context` (text) プロパティを定義。
- `lib/db_operations.py` の登録関数を更新。

### タスク2: 現場向けハイブリッド入力UIの構築
**目的:** 思考を止めない入力体験を実現する。
- Streamlitの `st.columns` や `st.button` を活用し、タイル状の感情・タグ選択画面を作成。
- `st.chat_input` もしくは音声入力コンポーネントによる詳細状況入力を統合。

### タスク3: 管理者向けインサイト・フィードの実装
**目的:** 現場の「今」を可視化する。
- Neo4jから最新のログを `ORDER BY date DESC` で取得し、カード形式で表示。
- AI相談が行われたログを強調表示（バッジ付与等）。

### タスク4: リスク予測エンジンの実装
**目的:** データの蓄積を「予兆」に変える。
- 特定期間（直近7日 vs 30日）の感情出現比率を計算するCypherクエリを作成。
- 閾値（30%増など）に基づき、管理画面にアラートを表示するロジックを実装。

### タスク5: 安全第一（Safety First）プロトコルの統合
**目的:** 事故を未然に防ぐ。
- 入力画面の常に下部に、当該クライアントの `:NgAction` をリスト表示。
- 重大な禁忌事項（LifeThreatening）は赤文字で強調。

## 3. 実装上の注意点（DRY / YAGNI / TDD）

- **DRY:** 感情ログの保存処理は既存の `register_support_log` を拡張し、重複を避ける。
- **YAGNI:** 初期段階では複雑な機械学習モデルは使わず、まずは「移動平均による変化点検知」というシンプルな統計手法を採用する。
- **TDD:** 予測エンジンについては、あらかじめ「異常値」を含むテストデータを定義し、正しくアラートが出ることをユニットテストで保証する。
